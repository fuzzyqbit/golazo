<objective>
Lock the web app to localhost-only binding as a defense-in-depth security control. Two layers of enforcement: (1) CLI-level via `next dev -H 127.0.0.1` and `next start -H 127.0.0.1` in `web/package.json` scripts (Next.js's documented hostname-binding flag — Context7-verified for Next 16.x); (2) runtime guard via `web/instrumentation.ts`'s `register()` hook that reads `process.env.HOST` at server startup and aborts with a `WEB-03`-tagged error if the value is non-loopback. The two layers cover different attack vectors: layer 1 blocks the operator from forgetting the flag in normal use; layer 2 blocks any caller that bypasses the script (e.g. `cd web && HOST=0.0.0.0 npx next dev` — which would otherwise inherit Next's default 0.0.0.0 binding).

Purpose: WEB-02 mandates exclusive loopback binding; WEB-03 mandates the HOST env-var guard. Both requirements demand a security-significant change to the dev/start surface AND a regression test that proves both layers fire. Splitting the hardening into its own plan (separate from the scaffold in Plan 02 and the fonts in Plan 04) keeps the security rationale auditable as an atomic change and lets the unit + integration tests stay tightly focused on the WEB-02/WEB-03 contract. File-disjoint with Plan 04 (Plan 04 touches `web/src/app/*` + new `web/src/theme/*` + new `web/src/fonts.ts` only) so this plan can run in parallel in Wave 3.

Output: Two-layer localhost-only enforcement: `-H 127.0.0.1` flag in dev/start scripts + runtime guard in instrumentation.ts. Unit suite covers the pure validator. Integration test spawns `next dev` in three scenarios (default + HOST=0.0.0.0 bare + HOST=0.0.0.0 composed via npm run dev) and pins both isolated-layer behaviour AND composed-operator-path behaviour via lsof + exit-code assertion. web/src/lib/hostGuard.ts (NEW):

  Pure module — no Next.js imports, no Node.js runtime side effects at
  module-load time (so unit tests can import it directly without spinning
  up a server context).

  export const LOOPBACK_HOSTS = ['127.0.0.1', 'localhost', '::1', '[::1]'] as const;

  export type LoopbackHost = typeof LOOPBACK_HOSTS[number];

  /**
   * Validate that the resolved HOST value is loopback-only.
   *
   * Allowed values (case-insensitive after trim):
   *   - undefined (Next.js's default behaviour applies — combined with the
   *     -H 127.0.0.1 flag in package.json's dev/start scripts, the effective
   *     bind is still loopback)
   *   - '' (empty string; treated the same as undefined)
   *   - any value in LOOPBACK_HOSTS
   *
   * Throws Error with message:
   *   `WEB-03: refusing to bind to non-loopback HOST '<host>'. Set HOST=127.0.0.1 (or unset) to start the web app.`
   *
   * The 'WEB-03' literal in the message is contractually pinned (test #6 in
   * hostGuard.test.ts and the integration test both assert it). Future
   * refactors MUST preserve this token so the error correlates with the
   * requirement.
   */
  export function validateHostBinding(host: string | undefined): void {
    const normalised = (host ?? '').trim().toLowerCase();
    if (normalised === '') return;
    if (LOOPBACK_HOSTS.some(h => h.toLowerCase() === normalised)) return;
    throw new Error(
      `WEB-03: refusing to bind to non-loopback HOST '${host ?? ''}'. Set HOST=127.0.0.1 (or unset) to start the web app.`
    );
  }

  Edge cases the function deliberately handles:
  - Whitespace-padded values (trim before compare) — operator typo defence
  - Mixed-case 'LocalHost' (lowercase before compare)
  - Empty string == undefined (Next.js sometimes injects empty HOST via env propagation)
  - '0.0.0.0' — the canonical bad value; rejected
  - '127.0.0.1:4173' — rejected (this is HOST+PORT, not HOST; Next.js uses -p separately)
  - LAN IPs (10.x, 192.168.x, 172.16-31.x) — all rejected because none are in LOOPBACK_HOSTS

web/instrumentation.ts (NEW):

  Next.js's documented startup hook (Context7-verified):
  https://nextjs.org/docs/app/api-reference/file-conventions/instrumentation

  /**
   * Next.js calls register() once on server startup (for each runtime that
   * boots — Node.js and Edge separately). We only want to validate HOST in
   * the Node.js runtime: Edge runtime doesn't process the same HOST env var
   * and aborting in the Edge bootstrap can cause confusing dev errors.
   */
  export async function register(): Promise<void> {
    if (process.env.NEXT_RUNTIME !== 'nodejs') return;
    const { validateHostBinding } = await import('./src/lib/hostGuard.js');
    validateHostBinding(process.env.HOST);
  }

  Notes:
  - File MUST be at `web/instrumentation.ts` (project-root convention; Next.js auto-detects). If web/tsconfig.json's `paths` or `rootDir` would relocate it, document the exception.
  - `await import(...)` (dynamic) instead of static `import` so the cost is only paid when register() actually runs — keeps cold-boot fast.
  - The `.js` extension on the dynamic import path is correct for ESM + bundler resolution; if Next.js / Turbopack rejects this, try `'./src/lib/hostGuard'` (no extension) per Next's documented file-convention import patterns.
  - If `validateHostBinding` throws, Next.js's startup proceeds in two modes:
    - dev: Next reports the error to stderr and exits the dev server
    - prod (`next start`): Next reports and exits
    Either way, the process terminates non-zero — the integration test pins this.

Locked decisions surfaced by this plan (record in SUMMARY):

  D-08: Two-layer enforcement (CLI flag + instrumentation guard).
        Rationale (defense in depth): the -H flag in package.json scripts
        is the primary defence — the operator's `npm run web:dev` always
        binds loopback. The instrumentation guard is the secondary defence
        — catches any direct `npx next dev` or `node ./web/.next/...`
        invocation that bypasses the script. WEB-03 explicitly mandates
        defense in depth.

  D-09: register() guards only Node.js runtime (NEXT_RUNTIME === 'nodejs').
        Rationale: Edge runtime has different HOST semantics and abort
        inside Edge bootstrap is confusing. Trade-off acknowledged: a
        production deployment that runs Edge-only would skip the guard.
        v2.0 ships only the dev server + `next start` (both Node.js); no
        Edge deployment path exists yet.

  D-10: Error message contractually pins the literal `WEB-03` token.
        Rationale: lets the integration test grep stderr for the
        requirement ID; lets the operator correlate the error with the
        requirements doc directly. Future refactors MUST preserve this
        token; the integration test enforces it.

  D-11: Tests live in `web/src/lib/hostGuard.test.ts` (co-located unit)
        AND `web/tests/host-binding.integration.test.ts` (integration).
        Rationale: the unit test pins the pure function; the integration
        test pins the end-to-end behaviour with a real spawn. Both are
        required for confidence — the unit test could pass while the
        instrumentation guard is silently disabled (e.g. wrong file
        path); the integration test could fail for unrelated reasons
        (port conflict, Next.js startup timeout) and the unit test
        isolates the pure logic in that case.

  D-11b: Integration test includes Scenario C (composed operator path —
        `HOST=0.0.0.0 npm run dev` with script-level `-H 127.0.0.1` flag).
        Rationale: A + B test each defense layer in isolation; without C,
        the composition (which is the actual operator-facing behaviour)
        was only smoke-tested manually. C automates the precedence proof:
        CLI flag wins over env guard, no security regression in the
        composed path. Asserts via lsof that the bind is 127.0.0.1 even
        when HOST=0.0.0.0 is set, because the dev script's `-H 127.0.0.1`
        takes precedence.

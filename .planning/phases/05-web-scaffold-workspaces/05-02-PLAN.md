    "compilerOptions": {
      "noEmit": true,
      "jsx": "preserve",
      "module": "ESNext",
      "moduleResolution": "bundler",
      "lib": ["ES2023", "DOM", "DOM.Iterable"],
      "rootDir": ".",
      "incremental": true,
      "tsBuildInfoFile": ".next/cache/tsbuildinfo",
      "plugins": [{ "name": "next" }],
      "paths": {
        "@/*": ["./src/*"]
      }
    },
    "include": ["next-env.d.ts", "src/**/*", "tests/**/*", ".next/types/**/*.ts"],
    "exclude": ["node_modules"]
  }

  Notes:
  - `extends: "../tsconfig.json"` reuses strict, noUncheckedIndexedAccess, target, esModuleInterop, skipLibCheck, forceConsistentCasingInFileNames, resolveJsonModule from root.
  - `moduleResolution: "bundler"` (NOT `NodeNext` — Next.js documents `bundler` for App Router projects to match Turbopack's import resolution; `NodeNext` would force `.js` extensions on every relative import inside the web tree, which is incompatible with the Next.js App Router file conventions where `import './globals.css'` and `import Component from './Component'` are the documented patterns).
  - `module: "ESNext"` (overrides root's `NodeNext` — see above).
  - `lib: ["ES2023", "DOM", "DOM.Iterable"]` — root's lib is `["ES2023"]` (Node-only); web needs DOM. Listing `ES2023` again here keeps the union explicit (TS merges arrays from extends).
  - `jsx: "preserve"` is the documented Next.js value (compiler emits JSX as-is; Next's webpack/Turbopack pipeline transforms it).
  - `plugins: [{ "name": "next" }]` enables Next's TypeScript plugin (route type checking, etc.).
  - `paths` maps `@/*` to `./src/*` — Next.js convention for absolute-style imports inside the web tree. Plans 04, 06, 07 will use `@/theme`, `@/lib/scanner`, etc.
  - `rootDir: "."` is the web subdir; web/'s own TS sources live under `web/src/` and `web/tests/`. The root tsconfig's `rootDir: "./src"` is a different scope (root TS sources under repo-root `src/`); the two configs are entirely separate compilation contexts.
  - The extends chain DOES inherit root's `noUncheckedIndexedAccess: true` and `strict: true` — Plan 04 + Phase 6+ web code MUST handle this (the same idiom the v1.0 src/ code uses).

Locked decisions surfaced by this plan (record in SUMMARY):

  D-04: tsconfig moduleResolution = "bundler" inside web/tsconfig.json.
        Rationale: Next.js App Router documents `bundler`; Turbopack expects
        it; using NodeNext would force `.js` extensions on every relative
        import in web/, which conflicts with Next.js conventions. Trade-off:
        web/ and root src/ use different moduleResolution strategies.
        Acceptable because they are different compilation contexts (web is
        bundled by Turbopack; src/ is compiled by tsc).

  D-05: Default Turbopack port = 4173. Rationale: avoid Next.js default 3000
        (commonly occupied during development); semantically signals "preview"
        (matches Vite's preview-port convention); fixed value avoids the
        "what port did it pick" cognitive load.

  D-06: `@golazo/cli` import paths use the `dist/` subpath (e.g.
        `@golazo/cli/dist/prepare/manifest.js`). No `exports` map added to
        root package.json. Rationale: keeps Plan 01 + Plan 02 small;
        defers the exports-map work until multi-import convenience justifies it.

  D-07: Cross-workspace type-import smoke lives in `web/tests/workspace-import.test.ts`
        (NOT in root `tests/`). Rationale: the test belongs to the web package;
        running it from web/ uses the web tsconfig (which has DOM lib + bundler
        moduleResolution), correctly simulating real web-code import patterns.

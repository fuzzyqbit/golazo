<sort_contract>
<!-- Pinned URL surface for UI-01. Plan 03's Server Component reads searchParams and calls parseListParams; Plan 03's client component updates URL via router.replace with serializeListParams output. -->

Sort key vocabulary: `date | opponent | result | kid`
Sort dir vocabulary: `asc | desc`
URL serialization: `?sort=<key>.<dir>` (single string, dot-separated)
Default: `sort=date.desc` (omitted from URL when default)
Kid filter vocabulary: `all | leo | mateo`
Default: `kid=all` (omitted from URL when default)

Result ordering for `result.asc`: `W → D → L` (best-to-worst); `result.desc` is reverse.
Rationale: 'W'/'D'/'L' lexicographic would put D before L which is fine, but operator-intuitive ordering for "sort by result ascending" is "best first" — pin the explicit mapping `{W: 0, D: 1, L: 2}` in code.

Invalid input handling: unknown sort key, unknown dir, unknown kid → silently fall back to the default. No throws, no error UI. parseListParams is called from a Server Component on every request — it MUST not throw on operator-typed nonsense in the URL bar.
</sort_contract>

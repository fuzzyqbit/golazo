<detail_view_layout>
<!-- Pinned for PLAY-02 — every section MUST render. -->

Top-down rendering order (CSS Modules per Plan 05-04 D-14):
1. **Back link** — `<Link href="/">← all episodes</Link>` (use serializeListParams from Plan 01 if you want to round-trip the previous filter, but a plain `/` link is acceptable for v1)
2. **Title section** — the rendered title from `renderTitle(templateInput)` displayed as `<h1>` in Cormorant Garamond Italic (var(--font-display)). This is the YouTube title string verbatim (includes U+00B7 MIDDLE DOT separators and U+2013 EN DASH score separator — those characters come from the template renderer; do not alter)
3. **Description block** — the rendered description from `renderDescription(templateInput)` displayed in a `<pre>` element with `white-space: pre-line` (preserves \n newlines from the template). Label font, foreground color.
4. **Manifest section** — fold-out details:
   - manifestHash (full sha256:... string), monospace label-font
   - Clip list: `<ol>` of `{filename}{spacer}{durationSec}s` — one `<li>` per manifest.clips entry. Render entries in the manifest order (already sorted by the scanner / manifest builder).
   - Music pick: if manifest.music present, render `Music: {track} ({durationSec}s)`; if absent, render `Music: (not set)`
   - Render block: if manifest.render present, render `Rendered: {renderedAt} → {episodePath}`; if absent, render `Render: (not yet rendered)`
5. **Publish section**:
   - If publish.json present + valid: render `videoId`, `watchUrl` as an external `<a target="_blank" rel="noopener noreferrer">`, `uploadedAt`, and a separate `YouTube Studio` link to `https://studio.youtube.com/video/<videoId>/edit`
   - If absent or invalid: render `<p>Not published yet.</p>` and a hint `Run \`golazo publish <folder>\` to upload.`

NO `'use client'` directive in this entire plan — the detail view is fully server-rendered. Phase 8 will add the `<video>` player as a separate concern; that may or may not require a small client island.

Typography intent matches Plan 03: Cormorant Garamond Italic for headings + opponent name; Inter for everything else; COLORS background/foreground/accentDefault; U+2013 EN DASH for scores.
</detail_view_layout>

<file_read_strategy>
<!-- manifestRead + publishRead helpers — keep them small + tested. -->

These wrap the synchronous reads + zod validation. The scanner module (web/src/lib/scanner.ts) already does this work for the list view; the detail view does it once per request for the selected row only.

`readManifestFromRow(row: EpisodeIndex): Manifest` — throws ManifestReadError when:
- The file does not exist at `path.join(row.absFolderPath, GOLAZO_DOT_DIR, MANIFEST_FILE_NAME)`
- JSON parse fails
- manifestSchema validation fails
The dynamic route catches ManifestReadError and treats it as notFound() (the row exists in cache but the on-disk file vanished between scan and request — extremely rare; degrade gracefully).

`readPublishFromRow(row: EpisodeIndex): PublishRecordDoc | null` — returns null when:
- The file does not exist
- JSON parse fails
- publishRecordSchema validation fails
Does NOT throw — publish.json is optional state. The presence of `row.publishVideoId !== null` is the cache's view of publish state; the on-disk read is the live truth (the watcher keeps these consistent, but a transient mismatch is acceptable).

Both helpers MUST import the schemas from @golazo/cli/dist/... — never redeclare them.
</file_read_strategy>

# Quick Task 260622-d47: OAuth OOB → 127.0.0.1 Loopback Redirect Summary

Replaced the deprecated Google OOB OAuth redirect (`urn:ietf:wg:oauth:2.0:oob`, blocked by Google in 2023) with the 127.0.0.1 loopback-IP flow Google requires for native/Desktop clients: `runAuth` now starts an ephemeral `node:http` server on `127.0.0.1:0`, bakes `redirect_uri=http://127.0.0.1:<port>` into the consent URL, and captures the authorization code from the browser redirect instead of prompting for a stdin paste.

## What Changed

- **`src/publish/oauth.ts`**
  - Removed `OOB_REDIRECT_URI` and the OOB file-header doc; the module header now documents the 127.0.0.1 loopback flow.
  - `createOAuth2Client`: no static redirect default — passes `opts.redirectUri` straight through, omitting the third `OAuth2()` arg when none is supplied (2-arg form). `loadToken` continues to build a client without a redirect.
  - New exported `captureLoopbackCode(): Promise<LoopbackCapture>` — creates a `node:http` server, `server.listen(0, '127.0.0.1')`, resolves once listening with `{ port, redirectUri, codePromise, server, close }`. The request handler:
    - `?code=…` → 200 close-tab HTML page, resolves `codePromise`, closes the server (releases the port).
    - `?error=…` → 200 page, rejects `codePromise` with `OAuthError({ field: 'consent', reason: 'authorization denied: <error>', … })`, closes the server.
    - other paths (e.g. `/favicon.ico`) → 204, no resolution.
    - Never logs the query string, code, or token bytes.
  - `runAuth` rewired: claims the loopback port FIRST, builds the client with `redirectUri: capture.redirectUri`, builds the consent URL, obtains the code via the injected `readCode` stub OR `defaultReadCode` (prints URL, awaits `capture.codePromise` — no stdin), exchanges, saves, returns. A `try/finally` calls `capture.close()` on success or throw so no listener leaks.
  - `defaultReadCode` now prints the consent URL + a "Waiting for the browser redirect" line and awaits the loopback capture (stdin/`node:readline` removed).
  - Preserved verbatim: the `readCode` injection seam on `RunAuthOpts`, and the `GOLAZO_OAUTH_MOCK=1` shim in `exchangeCode`.
- **`src/publish/oauth.test.ts`** — Added describe block `captureLoopbackCode (127.0.0.1 loopback redirect)` with cases A–D (bind+parse/close-tab, denial→OAuthError(field:consent), port-release after capture, never-logs the code). All 19 original cases (incl. the readCode-seam cases 15–17 and the GOLAZO_OAUTH_MOCK case 19) kept and green.
- **`src/cli/auth.integration.test.ts`** — Reworked off the OOB stdin paste. Happy-path + never-logs cases (1, 2) spawn the CLI, read stdout until the consent URL line appears, parse `redirect_uri` to recover `http://127.0.0.1:<port>`, then `http.get(.../?code=fake-code)` to simulate the browser redirect (primary approach — proved stable, fallback not needed). Error-path cases (3 unknown-kid, 4 missing-client-id) fail before the server starts, so the stdin paste was simply dropped.

## Deviations from Plan

**1. [Plan-ergonomics] `captureLoopbackCode` returns a Promise, not a sync object.**
- **Why:** `server.address()` returns `null` synchronously immediately after `server.listen()` — the socket isn't bound until the async `listening` event. To expose `port`/`redirectUri` reliably, `captureLoopbackCode()` returns `Promise<LoopbackCapture>` that resolves in the `listening` callback. The plan's Task 1 sketch wrote `const capture = captureLoopbackCode()` synchronously; the committed tests use `await captureLoopbackCode()` and `runAuth` awaits it. The plan explicitly left the shape "your call as long as Task 1 tests pass" — the returned object still carries `{ port, redirectUri, codePromise }`, satisfying the must-haves and key-links (`listen(0`, `127.0.0.1`).
- **Adjustment was made during GREEN** (the 4 unit cases were updated to `await`), which is why a small slice of the RED test file appears again in the GREEN commit.

**2. [Defensive] Attached a no-op `.catch()` to `codePromise` inside the helper.**
- The denial path can reject `codePromise` before the caller awaits it (e.g. in test B between the GET and the `expect(...).rejects`). A no-op catch prevents an unhandled-rejection crash; the caller still observes the rejection on its own `await`. Not in the plan text but a correctness requirement (Rule 2).

No other deviations. The `readCode` seam and `GOLAZO_OAUTH_MOCK=1` shim are preserved exactly.
